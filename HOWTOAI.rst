How to Use AI with os-specific-runner
======================================

*A practical guide for contributing to os-specific-runner using AI coding assistants*

``os-specific-runner`` is a focused, single-purpose GitHub Action written in JavaScript
(ES modules, Node.js 24). AI tools can meaningfully accelerate contributions, but every
contributor is responsible for understanding and owning the code they submit. Whether you
use GitHub Copilot, Cursor, Claude, or another assistant, this guide will help you
contribute effectively.

----

Core Principles
---------------

* **Human Oversight**: You are accountable for every line you submit. Never commit
  AI-generated code you cannot explain or debug yourself.
* **Quality Standards**: AI-assisted code must pass the same bar as hand-written
  code — tests, documentation, and consistency with existing patterns.
* **Transparency**: Note significant AI involvement in your PR description and describe
  how you validated the result.

Best Practices
--------------

✅ Recommended Uses
~~~~~~~~~~~~~~~~~~~

* Generating Jest test cases for ``lib.js`` helper functions
* Writing or updating RST/Markdown documentation
* Drafting additions to ``builtInShells`` and ``fileExtensions`` in ``lib.js``
* Refactoring ``index.js`` for readability without changing behaviour
* Understanding ``@actions/core`` and ``@actions/exec`` API patterns
* Scaffolding new ``action.yml`` input definitions and matching JS logic
* Writing ``*.yml`` workflow snippets that use the action

❌ Avoid AI For
~~~~~~~~~~~~~~~

* Changes to ``dist/index.js`` by hand — this file is always generated by
  ``ncc build index.js -m`` and must never be edited directly
* Shell command strings inside ``builtInShells`` without manually verifying
  behaviour across all three platforms (Linux, macOS, Windows)
* Any modification that embeds user-controlled data directly into a shell string
  without reviewing for injection risks
* Large restructuring of ``index.js`` without a clear rationale and full test coverage
* Code you do not fully understand

Workflow Tips
~~~~~~~~~~~~~

* Build and test after every meaningful change: ``npm install && npm test``
* Regenerate the bundle before opening a PR: ``npm run build``
* Study how ``formatShell``, ``fileExtensions``, and ``builtInShells`` work together
  in ``lib.js`` before asking AI to modify them
* Ask yourself: "Does this change break any of the three OS paths? Does it handle
  an empty ``working_directory`` correctly? Does the shell template still need ``{0}``?"
* Test shell additions locally on the target platform if you have access to it;
  otherwise rely on the CI matrix (``ubuntu-latest``, ``windows-latest``, ``macos-latest``)

Security Considerations
~~~~~~~~~~~~~~~~~~~~~~~

* The action writes user-supplied command strings directly to a temp file and executes
  them via the chosen shell — AI-generated workflow snippets that use this action
  must be reviewed for shell injection or unexpected side-effects
* Never paste real workflow secrets, tokens, or credentials into an AI chat prompt
* Review any AI-suggested shell flag changes (e.g., removing ``-eo pipefail``) carefully
  — those flags exist for good reasons
* Do not allow AI to introduce ``eval``-style patterns or dynamic shell construction
  beyond what is already in ``index.js``

Testing & Review
----------------

Before submitting AI-assisted code, confirm that:

* You understand every change in ``index.js``, ``lib.js``, and ``action.yml``
* All unit tests pass: ``npm test``
* The ``dist/index.js`` bundle is current: ``npm run build``
* Any new helper exported from ``lib.js`` has corresponding Jest tests in
  ``tests/index.test.js``
* The ``action.yml`` and the JavaScript code agree on input names, defaults, and types
* Documentation in ``README.md`` reflects any new inputs or behavioural changes

**Always get human review for:**

* Any change to how user-supplied command strings are written to disk or executed
* New shell templates added to ``builtInShells`` (cross-platform correctness matters)
* Additions or renames of ``action.yml`` inputs (breaking change for existing users)
* Changes to the temp-file path logic in ``getTempWorkingDir``
* Anything that touches the ``dist/`` build artefact workflow (``regenerate.yml``)
* Large refactors or anything you are unsure about

Understanding os-specific-runner's Architecture
-------------------------------------------------

Essential Concepts
~~~~~~~~~~~~~~~~~~

The action has three source files that each play a distinct role:

.. code-block::

   action.yml          ← declares inputs, defaults, and the Node.js 24 runtime
       │
       ▼
   index.js            ← entry point: reads inputs, detects OS, writes script to temp
       │                 file, and executes it via @actions/exec
       │  imports
       ▼
   lib.js              ← pure helpers: formatShell(), getTempWorkingDir(),
                         fileExtensions{}, builtInShells{}

       builds to
       ▼
   dist/index.js       ← single-file bundle produced by `ncc build index.js -m`
                         (this is what GitHub Actions actually runs)

**``action.yml``** is the contract between the action and its callers. It names every
input (``linux``, ``macos``, ``windows``, ``linuxShell``, ``macosShell``,
``windowsShell``, ``working_directory``), sets defaults, and points the runtime at
``dist/index.js``.

**``lib.js``** is intentionally free of side-effects so it is easy to unit-test in
isolation. All logic that can be expressed as a pure function lives here:

* ``formatShell(template, ...args)`` — replaces ``{0}``, ``{1}`` … placeholders in
  a shell command template with the provided arguments.
* ``getTempWorkingDir(working_directory)`` — returns a cross-platform absolute temp
  path using ``os.tmpdir()``.
* ``fileExtensions`` — maps shell names to required script-file extensions
  (e.g. ``pwsh`` → ``.ps1``).
* ``builtInShells`` — maps shell names to their invocation templates
  (e.g. ``bash`` → ``'bash --noprofile --norc -eo pipefail {0}'``).

**``index.js``** ties everything together at runtime:

1. Read ``working_directory`` input and create the temp directory.
2. Detect the OS with ``os.platform()`` (``'linux'``, ``'darwin'``, ``'win32'``).
3. Select the matching command and shell inputs.
4. Look up the file extension and shell template from ``lib.js``.
5. Write the command string to a UUID-named temp file.
6. Execute the formatted shell string with ``@actions/exec``.

Example 1: Adding a New Supported Shell (``fish``)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

**Step 1 — Explore existing patterns**

Read ``lib.js`` and ask AI: *"Following the pattern of the existing entries in*
``builtInShells`` *and* ``fileExtensions``, *what entries would I need to add for the*
``fish`` *shell? Fish scripts have no standard extension and the invocation is*
``fish <file>``."*

**Step 2 — Draft the change in ``lib.js``**

AI should produce something like the following additions:

.. code-block:: javascript

   // In fileExtensions — fish scripts have no required extension, so no entry needed.
   // (absence from the map means '' is used, which is correct)

   // In builtInShells:
   fish: 'fish {0}',

Verify the template contains ``{0}`` and matches the pattern of ``sh: 'sh -e {0}'``.

**Step 3 — Add a Jest test in ``tests/index.test.js``**

.. code-block:: javascript

   test('fish template formats correctly', () => {
       expect(formatShell(builtInShells['fish'], '/tmp/script'))
           .toBe('fish /tmp/script');
   });

   test('fish has no file extension', () => {
       expect(fileExtensions['fish']).toBeUndefined();
   });

**Step 4 — Update ``action.yml`` documentation comments** (if you are documenting
supported shells in a comment block) and update ``README.md`` to list ``fish`` under
the supported shells table.

**Step 5 — Validate**

.. code-block:: bash

   npm test        # all existing + new tests must pass
   npm run build   # regenerate dist/index.js

Note that ``fish`` must actually be available on the runner for the integration test
to pass. Mention this caveat in your PR.

Example 2: Fixing a Failing Jest Test
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

**Step 1 — Copy the failure output**

Run ``npm test`` and copy the full failure block, for example:

.. code-block:: text

   ● getTempWorkingDir › appends working_directory to tmpdir

     expect(received).toBe(expected)

     Expected: "/tmp/my_dir/sub"
     Received: "/tmp/carlkidcrypto/os-specific-runner/my_dir/sub"

**Step 2 — Ask AI with context**

Paste the failure and the current implementation of ``getTempWorkingDir`` from
``lib.js``, then ask: *"The test expects the path to be joined directly under*
``tmpdir()``, *but the implementation prefixes* ``carlkidcrypto/os-specific-runner``
*when* ``working_directory`` *is non-empty. Which one is correct — the test or the
implementation? Show me the minimal fix."*

**Step 3 — Understand before applying**

Read the AI's explanation. Decide whether the test expectation or the implementation
is wrong by checking related tests and the ``README.md`` description of
``working_directory``. Do not blindly apply the fix.

**Step 4 — Apply the fix to the right file**

If the test is wrong, correct the expected value in ``tests/index.test.js``.
If the implementation is wrong, fix ``getTempWorkingDir`` in ``lib.js``.

**Step 5 — Verify**

.. code-block:: bash

   npm test        # the previously failing test must now pass; no regressions
   npm run build   # always regenerate after touching lib.js

Example 3: Adding a New Action Input (``timeout``)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

**Step 1 — Define the input in ``action.yml``**

Ask AI: *"Add an optional* ``timeout`` *input to this* ``action.yml`` *that accepts a
number of seconds, defaults to empty string (meaning no timeout), and has a clear
description."*

Expected AI output to add under ``inputs:``:

.. code-block:: yaml

   timeout:
     description: >
       Maximum number of seconds to allow the command to run.
       Leave empty (default) for no timeout.
     required: false
     default: ""

**Step 2 — Read the input in ``index.js``**

Ask AI: *"Show me where in* ``index.js`` *I should read the new* ``timeout`` *input
using* ``getInput('timeout')``, *parse it as an integer, and pass it to*
``@actions/exec`` *via its options object. Show only the minimal diff."*

Review the suggested change and confirm:

* ``getInput`` is called after the OS detection block
* The integer parse guards against non-numeric input (``parseInt`` + ``isNaN`` check)
* The ``exec`` options object already used (or easily extended) in ``index.js`` is
  where the timeout is applied

**Step 3 — Add a unit test**

Because timeout logic in ``index.js`` involves ``@actions/exec`` (a side-effectful
call), the most practical test is an integration one. Add a step to
``integration_tests.yml`` using a fast ``sleep`` command and a short timeout, and
confirm CI stays green.

If you extract a pure helper function (e.g., ``parseTimeout(raw)``), add it to
``lib.js`` and test it in ``tests/index.test.js``:

.. code-block:: javascript

   describe('parseTimeout', () => {
       test('returns undefined for empty string', () => {
           expect(parseTimeout('')).toBeUndefined();
       });
       test('returns integer for numeric string', () => {
           expect(parseTimeout('30')).toBe(30);
       });
       test('returns undefined for non-numeric string', () => {
           expect(parseTimeout('abc')).toBeUndefined();
       });
   });

**Step 4 — Update documentation**

Add the new input to the inputs table in ``README.md`` and rebuild:

.. code-block:: bash

   npm test
   npm run build
